# 77번 — 팀원 상호 피드백 리뷰 & 프로젝트 보완 문서 · 소울나루

77번 — 팀원 상호 피드백 리뷰 & 프로젝트 보완 문서 · 소울나루 **# 최근 생성 문서 팀원 상호 피드백 리뷰 & 프로젝트 보완 문서
대상 문서:** 59~74번 최근 생성 문서  |  **검토 방식:** PM·개발·QA·아트·기획/AI심리 관점 상호 리뷰  |  **목적:** Sprint 5 진입 전 문서 정합성 및 실행 가능성 보완

## 1. 리뷰 요약
**강점** FB 항목과 Sprint 5 목표가 문서 번호별로 빠르게 추적 가능하다.

**보완점** 일부 문서는 코드 반영 완료와 명세 완료가 혼재되어 있어 상태 정의가 필요하다.

**실행 리스크** IAP, Onboarding, Settings가 동시에 진행되어 QA 집중 구간이 5/4~5에 몰린다.

**개선 방향** 각 문서에 “완료 기준/검증 담당/증거 산출물”을 명시한다.

## 2. 팀별 상호 피드백

| 리뷰 팀 | 피드백 | 보완 조치 |
| --- | --- | --- |
| 개발팀 | 73번 결제 기술 명세는 구현 코드가 충분하나 RevenueCat API Key 보관 위치와 빌드 환경 분리가 필요 | 환경별 키 분리(.env/ScriptableObject), 개발/운영 로그 레벨 분리 추가 |
| QA팀 | 70·73번 IAP QA가 5/4~5에 집중되어 실패 시 회복 시간이 부족 | 5/2부터 Sandbox smoke test 선행, 5/4는 E2E 확정 테스트로 운영 |
| 아트팀 | 74번 씬 명세에 이미지 방향은 있으나 최종 톤앤매너 예시 부족 | 76번 예시 이미지 4종 제작 및 문서 참조 추가 |
| 기획·AI심리팀 | FB-16/17은 Sprint 5 P2이나 심리 안전과 직접 연결되어 우선순위 재검토 필요 | 위기 관련 문구는 Sprint 5 내 최소 검토 완료, 구현은 Sprint 6로 분리 가능 |
| PM팀 | 69~74번 문서가 빠르게 생성되어 index 내 탐색성이 중요 | 카드 UI 개선 및 검색/섹션 접기 유지 |

## 4. 보완 결론
최근 문서는 양적으로 충분하지만, Sprint 5 실행 관점에서는 상태 정의와 QA 증거 산출물이 가장 중요하다. 따라서 5/1 이후 생성되는 문서는 반드시 “검증 담당, 증거 산출물, 완료 기준”을 포함한다.
